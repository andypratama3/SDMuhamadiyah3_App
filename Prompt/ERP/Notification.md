# ERP — Notification Module

## Purpose

The cross-cutting delivery mechanism (in-app + push) for events generated by other modules
(Announcement, Payment, Attendance, PPDB, ReportCard) — Notification does not originate content, it
delivers events from other domains.

## Domain Model

```kotlin
data class NotificationItem(
    val id: String,
    val recipientId: String,
    val type: NotificationType,
    val title: String,
    val body: String,
    val deepLinkDestination: String,   // serialized Destination, see `11-NAVIGATION.md`
    val createdAt: Instant,
    val readAt: Instant?,
)
enum class NotificationType {
    AnnouncementPublished, PaymentReceived, InvoiceOverdue, InvoiceDueSoon,
    AttendanceAlert, ReportCardReleased, PpdbStatusChanged, MessageReceived, ApprovalRequested,
}

data class NotificationPreference(
    val userId: String,
    val type: NotificationType,
    val pushEnabled: Boolean,
    val inAppEnabled: Boolean,
)
```

## Business Rules

1. **Every `NotificationItem` carries a deep link** (`11-NAVIGATION.md` typed `Destination`,
   serialized) — tapping a notification always lands the user precisely on the relevant
   screen/record, never just the generic app home.
2. **Per-type opt-out is allowed**, except for `Urgent`-priority Announcements and any
   `ApprovalRequested` notification relevant to the recipient's role responsibilities (Principal
   cannot mute approval requests — operational necessity overrides preference).
3. **In-app notification badge counts** are derived from `readAt == null` items for that recipient —
   never a separately maintained counter that could drift from the underlying unread set.
4. **Push delivery is fire-and-forget from the client's perspective** — actual push infrastructure
   (FCM/APNs) is a backend concern; the client's job is to register a device token
   (`expect/actual PushTokenProvider` per `21-KMP-STANDARDS.md`) and handle incoming
   payloads/deep-link navigation when tapped.

## Screens

- Notification Center (all roles) — chronological list, unread-first, swipe-to-mark-read, see
  `14-LIST-STANDARDS.md`.
- Notification Settings — per-type toggle, see `13-FORM-STANDARDS.md`.

## Permission Matrix

Notifications are inherently personal/scoped — every role only ever sees `NotificationItem`s where
`recipientId` matches their own user ID. There is no cross-role visibility of others' notifications.

## Audit Requirements

Not independently audited beyond standard delivery logging (for debugging delivery failures) — the
*source event* (e.g. the Payment or Approval itself) carries the audit requirement per its own
module's spec.
